## 1. 과제 개요

### 1.1 목적
소프트웨어 개발 현장에서 기획자/고객이 보낸 메일을
개발자가 바로 활용할 수 있는 요구사항 문서로
자동 변환하는 LLM 프롬프트를 설계한다.

### 1.2 기능 요구사항
- 메일 입력 → 기능/비기능/미확인 3분류 자동 정리
- 모호한 입력 시 추정 금지 및 되묻기
- 수정/추가 요청 시 변경 항목 추적 가능
- 빈 입력 등 엣지케이스 안내 처리

### 1.3 사용 환경
- 플랫폼: 웹 (ChatGPT, Claude, Gemini)
- 사용자: 소프트웨어 개발자
- 출력 형식: 불릿 리스트 (공식 문서 톤)

## 2. 업무 과업 및 문제 정의

### 2.1 업무 과업
| 항목 | 내용 |
|------|------|
| 과업명 | 메일 기반 요구사항 정리 |
| 입력 | 기획자/고객 메일 전문 |
| 출력 | 기능/비기능/미확인 불릿 리스트 |
| 타겟 독자 | 소프트웨어 개발자 |
| 톤 | 공식 문서 |

### 2.2 문제 정의
- 기획 메일은 구어체, 감정 표현, 누락 정보가 많아
  개발자가 그대로 사용하기 어렵다.
- 수동 정리 시 담당자마다 형식이 달라 일관성이 없다.
- 미확인 사항을 놓치면 개발 착수 후 요구사항 변경이 발생한다.

### 2.3 해결 방향
- LLM 프롬프트로 메일을 구조화된 요구사항 문서로 변환
- 미확인 사항을 자동으로 추출하여 담당자 확인 요청
- 페르소나(REMI) 적용으로 출력 톤 일관성 확보

### 3.2 비교에 사용한 동일 입력

세 모델 모두 아래 **동일한 입력**을 사용하여 평가하였다.

```text
[과업]      요구사항 정리
[입력 메일]
안녕하세요, 기획팀 김민준입니다.
회원가입 페이지에 소셜 로그인 기능을 추가해 주세요.
- 카카오, 구글 로그인 지원
- 기존 이메일 회원가입도 유지
- 로그인 후 메인 페이지로 자동 이동
마감은 이번 달 말입니다.
[타겟 독자] 개발자
[출력 형식] 불릿 리스트 (기능/비기능/미확인 3분류)
[톤]        공식 문서
[금지 항목] 인사말, 이모지, 헤더, 팁 코멘트
```

> 위 입력은 GPT-4o mini, Claude Haiku, Gemini 1.5 Flash 세 모델에
> 동일하게 입력하였으며, 추가 설명이나 사전 안내 없이 단독 입력으로 평가하였다.

### 3.3 평가 기준 및 결과

| 톤 일관성 | 5 | 4 | 4 | Claude·Gemini는 일부 문장에서 부드러운 구어체 어미가 섞여 "공식 문서 톤" 지시가 약하게 흔들림 |

| **합계** | **25** | **19** | **16** | |

### 3.4 선정 모델
- **GPT-4o mini**
- 선정 근거 (요약)
  - 정확성, 형식, 톤 모든 항목에서 5/5 달성
  - 모호한 입력 시 추정 없이 되묻기 일관 수행
  - 미확인 사항 추출 정밀도가 가장 높음

**업무 과업 특성과 연결한 선정 이유(서술)**

"메일 기반 요구사항 정리"라는 과업은 일반적인 대화형 챗봇 과업과 성격이 다르다. 결과물이 개발 착수의 근거 자료로 그대로 쓰이기 때문에, 문장이 매끄러운 것보다 **① 형식이 매번 동일해야 하고(개발자가 습관적으로 같은 위치에서 같은 정보를 찾을 수 있어야 함), ② 확실하지 않은 내용을 확정처럼 말하면 안 되며(잘못된 요구사항으로 개발이 진행되는 리스크), ③ 정보가 부족하면 반드시 되물어야 한다(추측성 개발 착수 방지)**는 세 가지가 다른 무엇보다 중요하다.

3.3의 평가축은 임의로 고른 것이 아니라 이 세 가지 요구를 그대로 지표화한 것이다 — "형식 일관성"은 ①을, "미확인 사항 추출"과 "엣지케이스 대응"은 ②·③을 직접 측정한다. GPT-4o mini가 이 세 축 모두에서 최고점을 받았다는 것은 단순히 "더 똑똑하다"는 의미가 아니라, **이 업무 과업이 요구하는 안전장치(추정 금지, 형식 고정, 확인 질문)를 프롬프트 지시만으로 가장 안정적으로 따랐다**는 뜻이다. Claude Haiku·Gemini 1.5 Flash도 문장 품질 자체는 준수했지만, 반복 지시(3분류 형식, 태그 규칙)를 매 턴 동일하게 지키는 데서 편차가 있었고, 이는 요구사항 문서처럼 "반복 재현성"이 핵심인 과업에서는 치명적인 단점이 된다. 따라서 GPT-4o mini 선정은 점수 총합이 아니라, **과업이 요구하는 안전성·형식 준수 특성과 모델의 강점이 정확히 일치했기 때문**이라는 것이 핵심 근거다.

**구체적 업무 시나리오로 본 선정 이유**

- 6장 Turn 3처럼 "결제 기능 추가해 주세요. 급합니다."처럼 정보가 극히 부족한 메일이 실제로 자주 들어온다. 이때 결제 수단·PG사·기한을 모델이 임의로 채워 넣으면, 개발팀은 존재하지 않는 요구사항을 근거로 스프린트를 계획하게 되고, 이는 실제 개발 리소스 낭비로 직결된다. GPT-4o mini는 이 상황에서 4가지 항목을 모두 되물었고(엣지케이스 대응 5점), Gemini 1.5 Flash는 테스트 과정에서 예시성 요구사항을 임의로 채워 넣는 경향을 보였다(3점) — 이 차이가 곧 "실제 스프린트 계획 오류를 막을 수 있는가"의 차이다.
- 만약 이 과업이 "카피라이팅"이나 "블로그 초안 작성"처럼 창의성·표현력이 우선인 과업이었다면, 톤의 풍부함이나 표현 다양성이 높은 모델(Gemini 계열이 상대적으로 강점을 보이는 영역)을 선택했을 것이다. 즉 이번 선정은 "GPT-4o mini가 항상 가장 좋은 모델"이라는 결론이 아니라, **"확정하지 않고 되묻는 보수적 태도"가 최우선 가치인 이 특정 업무 과업에서만 GPT-4o mini가 최적**이라는, 과업 종속적 결론이다.

## 4. 페르소나 정의 및 최종 프롬프트

### 4.1 페르소나 정의

| 항목 | 내용 |
|------|------|
| 이름 | REMI |
| 역할(직무) | 소프트웨어 개발팀 소속 요구사항 분석가(Business Analyst) |
| 전문 분야 | 비정형 메일/구어체 텍스트를 기능/비기능/미확인 요구사항으로 구조화하는 것 |
| 말투 | 공식 문서체, 존댓말, 감정 표현·구어체 어미 배제 |
| 금지 사항 | 인사말, 이모지, 헤더, 팁 코멘트 출력 금지 / 메일 원문에 없는 내용 추정·확정 금지 |
| 우선순위 | 정확성 > 친절함 — 모호한 입력에는 부드럽게 넘기지 않고 반드시 되물어 확인한다 |

### 4.2 단계적 추론 유도 적용 전/후 비교 (v1 vs v2)

동일한 Turn 1 입력(소셜 로그인 요청 메일)에 대해, 프롬프트 기법에 따른 결과 차이를 비교하였다.

**v1 (일반 지시 / Zero-shot)**
```text
아래 메일을 분석해서 요구사항을 정리해줘.

[입력 메일]
안녕하세요, 기획팀 김민준입니다.
회원가입 페이지에 소셜 로그인 기능을 추가해 주세요.
- 카카오, 구글 로그인 지원
- 기존 이메일 회원가입도 유지
- 로그인 후 메인 페이지로 자동 이동
마감은 이번 달 말입니다.
```

v1 결과(문제점):
```text
알겠습니다! 요구사항 정리해드릴게요~
- 카카오/구글 소셜 로그인 추가
- 기존 이메일 회원가입 유지
- 로그인 후 메인 이동
- 마감: 이번 달 말

이 정도면 충분할 것 같아요!
```
→ 인사말·구어체·이모지성 표현 포함, 기능/비기능/미확인 3분류 미준수, 미확인 사항(OAuth 키 주체, 계정 통합 정책 등) 전혀 도출되지 않음.

**v2 (단계적 추론 유도)**
```text
아래 메일을 분석할 때 다음 순서로 내부적으로 검토한 뒤 결과만 출력해줘.
(내부 검토 과정은 출력하지 말고, 최종 결과만 아래 형식으로 제시할 것)

1) 먼저 메일에서 확정적으로 언급된 사실만 골라낸다.
2) 개발 착수에 영향을 주는데 메일에 명시되지 않은 항목을 누락 질문(미확인 사항)으로 추출한다.
3) 위 내용을 [기능 요구사항] / [비기능 요구사항] / [미확인 사항] 3분류 불릿 리스트로만 출력한다.
   (인사말, 이모지, 헤더, 팁 코멘트 금지 / 메일에 없는 내용 추정 금지)

[입력 메일]
(Turn 1과 동일)
```

v2 결과: → 6장 Turn 1의 REMI 출력과 동일 (기능/비기능/미확인 3분류, 태그 포함, 4개 미확인 사항 도출)

**비교 결론**: v1은 형식·미확인 사항 추출이 불안정한 반면, v2는 "먼저 사실 확인 → 누락 질문 도출 → 정해진 형식으로만 출력"이라는 단계를 프롬프트에 명시함으로써 결과 품질과 일관성이 뚜렷하게 향상되었다. 단, 내부 추론 과정 자체는 출력에 노출하지 않고 "최종 결과만 정해진 형식으로 제시"하도록 규칙을 두어, 불필요한 장문의 사고 과정이 사용자에게 그대로 드러나지 않게 하였다.

### 4.3 최종 프롬프트 (v1.3) 전문 — 시스템 프롬프트 / 유저 프롬프트 분리

실제 배포 시(9.1 Claude Projects) 아래 두 블록은 서로 다른 위치에 들어간다. **[시스템 프롬프트]는 프로젝트의 "Custom Instructions"에 고정 등록**하고, **[유저 프롬프트/입력 템플릿]은 매 대화마다 사용자가 메일 원문과 함께 채워 넣는 부분**이다. 하나의 텍스트처럼 보이지만 역할이 다르므로 아래와 같이 물리적으로 분리해 관리한다.

**① 시스템 프롬프트 (역할 고정 · Custom Instructions에 등록)**

```text
당신은 REMI입니다.
소프트웨어 개발팀의 요구사항 분석 전문가로,
메일을 분석하여 개발자가 바로 사용할 수 있는
공식 문서 형식의 요구사항 목록을 작성합니다.

[출력 규칙]
- 메일 원문 기반 항목: (메일 원문 기준) 태그 추가
- 담당자 확인 항목: (확인 필요) 태그 추가
- 수정/추가 시: 해당 섹션에 (업데이트) 태그 추가

[미확인 사항 기준]
- 개발 착수에 직접 영향을 주는 항목만 포함
- 사소한 UI 문구, 디자인 세부사항은 제외

[금지 항목]
- 인사말, 이모지, 헤더, 팁 코멘트 출력 금지
- 메일에 없는 내용 추정 금지

[엣지케이스 처리]
- 입력 메일이 비어있을 경우: 입력 요청 안내 메시지 출력
- 정보가 부족할 경우: 추정하지 말고 확인 항목을 질문
- 범위 외 항목 확정 시: [범위 외 항목] 섹션에 별도 기록
```

**② 유저 프롬프트 / 입력 템플릿 (매 대화마다 사용자가 채워서 입력)**

```text
[과업]      요구사항 정리
[입력 메일] (분석할 메일 전문을 여기에 붙여넣기)
[타겟 독자] 개발자
[출력 형식] 불릿 리스트 (기능/비기능/미확인 3분류)
[톤]        공식 문서
[금지 항목] 인사말, 이모지, 헤더, 팁 코멘트
```

**분리하는 이유**: 시스템 프롬프트(①)는 REMI의 정체성과 판단 규칙(추정 금지, 태그 규칙, 엣지케이스 대응)을 담당하며 대화가 몇 번을 반복되든 변하지 않는다. 반면 유저 프롬프트(②)는 매번 다른 메일 원문이 들어오는 가변 입력이다. 이 둘을 하나의 텍스트로 섞어 매번 전체를 재입력하면 ① 반복 입력 비용이 크고 ② 규칙이 매번 미세하게 다르게 입력될 위험이 있으므로, 배포 시에는 반드시 ①은 프로젝트 지침에, ②는 대화창 입력으로 분리해서 운영한다.

## 6. 10턴 대화 로그

### 6.1 시나리오 설명

## 7. 환각 검증 설계

### 7.1 검증 목적 및 환각의 정의

REMI가 메일 원문에 없는 내용을 임의로 추가하거나
불확실한 정보를 확정된 것처럼 출력하는
**환각(Hallucination) 현상**이 발생하는지 검증한다.

**환각을 "사실 오류"로 한정하는 이유**

본 프로젝트에서 환각은 "사실/수치/정책처럼 정답 또는 검증 가능한 근거가 있어야 하는 항목에서, 근거 없이 틀린 정보를 확정적으로 서술하는 것"으로 정의한다. 예를 들어 미확인 사항으로 남겨야 할 "PG사"를 REMI가 임의로 "토스페이먼츠"라고 확정해버리면, 이는 실제로 존재하지 않는 계약·설정을 사실인 것처럼 서술한 것이므로 개발팀이 그대로 믿고 잘못된 시스템을 만들 위험이 생긴다. 즉 REMI의 출력물은 "그 자체가 개발 착수의 근거 자료"로 쓰이기 때문에, 검증 가능한 사실 영역에서의 오류는 업무상 실질적 피해로 이어진다.

반면 이 요구사항 정리 과업에는 "창의적 생성"이 필요한 지점이 원천적으로 없다 — REMI는 메일 내용을 재구성할 뿐, 새로운 아이디어나 표현을 만들어내는 역할이 아니다. 따라서 "허구임을 명시하면 되는" 창의적 글쓰기의 예외 규정은 이 과업에는 해당하지 않으며, REMI의 모든 출력은 예외 없이 "메일 원문 기준" 또는 "확인 필요" 둘 중 하나로만 분류되어야 한다는 것이 설계 원칙이다. 이 때문에 5개 검증 질문(7.3)도 전부 "검증 가능한 사실 영역"(기능 존재 여부, 정책 확정 여부, 수치, 요청 범위, 대화 문맥)에서만 설계했고, 창의적 표현의 자유도를 검증하는 항목은 의도적으로 포함하지 않았다.

**환각과 창의적 생성을 구분하는 명시적 기준**

| 구분 | 정의 | 판정 가능 여부 | REMI 과업 해당 여부 |
|------|------|----------------|---------------------|
| 환각(사실 오류) | 정답 또는 검증 가능한 근거가 있는 질문(정책, 수치, 계약, 범위 등)에서 근거 없이 틀린 정보를 확정적으로 서술 | 메일 원문·이전 대화와 대조하여 참/거짓 판정 가능 | **해당함** — REMI의 모든 출력은 개발 착수 근거로 쓰이므로 검증 대상 |
| 창의적 생성 | 정해진 정답이 없는 영역(소설, 카피, 브레인스토밍 등)에서 새로운 표현·아이디어를 만들어내는 것 | 원천적으로 참/거짓 판정 불가 (허구임을 명시하면 허용) | **해당 없음** — REMI는 메일 재구성만 수행, 새로운 아이디어 생성 기능 자체가 없음 |

이 표가 의미하는 바는, REMI에게는 "창의성을 발휘해도 되는 예외 영역"이 설계상 존재하지 않는다는 것이다. 따라서 REMI의 모든 출력 문장은 "메일 원문에서 검증 가능한가?"라는 단일 기준으로만 판정하면 되고, 이것이 곧 7.3의 5개 검증 질문이 전부 사실 검증형으로만 구성된 이유다.

### 7.2 시나리오 설명
아래 5개의 검증 질문은 각각 다른 환각 유형을 타겟으로 설계하였다.
- **임의 추가형**: 메일에 없는 항목을 스스로 생성하는지 확인
- **확정 오류형**: 미확인 사항을 확정된 것처럼 서술하는지 확인
- **범위 초과형**: 요청 범위를 벗어난 내용을 포함하는지 확인
- **수치 조작형**: 메일에 없는 수치/기한을 임의로 생성하는지 확인
- **문맥 혼용형**: 이전 대화의 내용을 현재 메일에 잘못 적용하는지 확인

### 7.3 검증 질문 및 Pass/Fail 기준

**Pass 기준의 공식 정의 (아래 5개 검증 항목 모두 이 기준을 따름)**

- **Pass 기준 A**: 검증 가능한 사실(메일에 실제로 적힌 내용)에 대해서는 정확한 답을 제시하고, 그 근거를 함께 밝힌다. (예: "메일에 명시된 A만 기능 요구사항으로 출력")
- **Pass 기준 B**: 메일에 명시되지 않아 답을 알 수 없는 사실에 대해서는 임의로 추측하지 않고 "확인 필요"로 명시하며, 어떤 정보가 더 필요한지 확인 절차를 제안한다.
- **Fail 기준**: 위 A/B 중 어느 쪽도 아닌 경우 — 근거 없는 내용을 사실처럼 단정하거나, 불명확한 부분을 확인 질문 없이 임의로 채워 넣으면 Fail.

각 검증 항목은 아래처럼 A/B 중 어떤 기준으로 Pass를 판정하는지 명시한다.

### 7.4 종합 Pass/Fail 기준

개별 항목의 Pass/Fail은 7.3에서 정의한 **Pass 기준 A(정답+근거 제시) / Pass 기준 B(확인 필요 명시+확인 절차 제안)** 중 하나라도 충족하면 Pass, 둘 다 충족하지 못하면(근거 없는 단정 또는 확인 질문 없는 임의 추정) Fail로 판정한다. 5개 항목의 개별 판정 결과를 종합하는 등급은 아래와 같다.

| 등급 | 기준 |
|------|------|
| **Pass** | 5개 검증 항목 중 5개 모두 통과 |
| **조건부 Pass** | 5개 중 4개 통과, 실패 항목이 수치 조작형 1개에 한함 |
| **Fail** | 5개 중 2개 이상 실패, 또는 확정 오류형/임의 추가형 실패 |

**판정 기준을 이렇게 설계한 이유**

다섯 가지 검증 유형은 업무상 위험도가 서로 다르기 때문에, 실패 개수만이 아니라 "어떤 유형이 실패했는가"를 함께 봐야 한다.

- **확정 오류형·임의 추가형은 1건만 실패해도 즉시 Fail**로 처리한다. 이 두 유형은 REMI가 "없는 사실을 있는 것처럼" 만들어내는 경우로, 개발팀이 이를 그대로 믿고 존재하지 않는 정책·기능을 개발에 반영할 수 있는 가장 직접적인 손해로 이어진다. 따라서 허용 오차 없이 즉시 실패로 판정한다.
- **수치 조작형은 상대적으로 조건부 허용**을 두었다. "빠르게"처럼 원래 모호한 표현을 수치로 못박는 것은 문제지만, 담당자가 리뷰 과정에서 상대적으로 발견하기 쉬운 오류(구체적 숫자가 눈에 띄기 때문)이므로, 다른 4개 항목이 모두 통과했다면 이 1건에 한해 "조건부 Pass"로 완화했다.
- **범위 초과형·문맥 혼용형**은 확정 오류형만큼 치명적이지는 않지만(존재하지 않는 사실을 만드는 것이 아니라 관련 없는 기존 사실을 잘못 가져오는 것), 반복되면 문서 신뢰도를 해치므로 2건 이상 실패 시 전체를 Fail로 처리한다.

즉 이 등급표는 "몇 개 맞았는가"가 아니라 "어떤 실패가 실제 업무에서 더 큰 피해로 이어지는가"를 기준으로 가중치를 다르게 준 것이며, 이는 7.1에서 정의한 "환각 = 검증 가능한 사실 영역에서의 근거 없는 확정"이라는 원칙과 일관되게 연결된다.

### 9.1 보너스 1 — 나만의 봇 배포 (재사용성 강화)

최종 프롬프트(v1.3, 4장)는 Claude.ai의 **Projects(프로젝트)** 기능으로 배포하여 재사용 가능한 형태로 만든다. Projects는 무료 Claude 계정에서도 최대 5개까지 사용할 수 있어 별도 결제 없이 배포가 가능하다.

**배포 절차**

| 단계 | 내용 |
|------|------|
| 1 | claude.ai 접속 → 왼쪽 사이드바 "Projects" → "+ New Project" 클릭 |
| 2 | 프로젝트 이름 지정 (예: "메일 요구사항 정리 REMI") |
| 3 | 프로젝트 지식(Project Knowledge)에 본 보고서(4.1 페르소나, 4.3 최종 프롬프트, 5장 Few-shot 예시) 업로드 |
| 4 | "Set project instructions"에 4.3의 v1.3 프롬프트 전문을 그대로 붙여넣고 저장 |
| 5 | 프로젝트 내에서 새 대화를 시작하면, 별도 설명 없이 메일 본문만 붙여넣어도 REMI가 3분류 요구사항 문서를 즉시 출력 |

**재사용성 확인**: 프로젝트 지침(Custom Instructions)에 등록해두면 이후 개설하는 모든 대화에 v1.3 프롬프트가 자동 적용되므로, 매번 프롬프트 전문을 재입력할 필요 없이 메일 원문만 붙여넣는 것으로 6장과 동일한 방식의 산출물을 반복 재현할 수 있다.

> 참고: 본 배포는 문서화 목적의 설계이며, 실제 배포 스크린샷·프로젝트 링크는 별도 첨부 가능.

## 10. 심층 인터뷰

과제 목표에서 요구하는 "학습자가 스스로 설명할 수 있어야 하는" 항목 중, 표나 규칙만으로는 드러나지 않는 설계 판단의 이유를 서술형으로 정리한다.

### 10.1 비용 제약(무료 모델 사용) 시 프롬프트·프로세스를 어떻게 바꾸었는가

3종 모델은 모두 웹 UI 무료판으로 테스트하였다(3.1). 무료판은 API처럼 temperature 등 생성 파라미터를 직접 낮춰 "일관성 모드"로 고정할 수 없고, 요청 횟수 제한 때문에 동일 조건 반복 테스트 횟수도 제한적이다. 이 제약을 프롬프트 설계로 보완한 지점은 다음과 같다.

- **파라미터로 못 잡는 일관성을 규칙으로 강제**: temperature를 낮출 수 없으므로, 대신 4.3 최종 프롬프트의 [출력 규칙]에 "메일 원문 기반 항목은 (메일 원문 기준) 태그, 확인 필요 항목은 (확인 필요) 태그"처럼 출력 형식을 토큰 단위로 못박았다. 모델이 매번 조금씩 다른 표현을 쓰더라도, 분류 체계와 태그 규칙만은 프롬프트 텍스트로 고정한 것이다.
- **반복 테스트 대신 유형별 1회 테스트로 설계**: API처럼 같은 입력을 수십 번 돌려 통계적으로 안정성을 검증하기 어려우므로, 7.3처럼 환각 유형을 5가지로 나누어 "각 유형당 1개의 대표 케이스"로 검증 범위를 압축했다. 양보다 유형 커버리지를 우선한 것이 무료판 제약에 대한 실질적 대응이다.
- **무료판 사용 제한을 문서에 명시**: 3.1에 "무료 버전 사용 제한 사항"을 별도로 기록해, 이 평가 결과가 유료판/API 조건에서는 달라질 수 있음을 재현자가 알 수 있게 했다.

**품질을 유지하기 위해 실제로 바꾼 프롬프트/프로세스 기법 (구체)**

무료 모델은 지시를 놓치는 빈도가 유료/최신 모델보다 높기 때문에, 아래처럼 "모델의 추론 능력에 기대는 대신 프롬프트 구조로 강제하는" 방향으로 설계를 바꾸었다.

1. **규칙 반복·명시화**: [출력 규칙], [금지 항목], [엣지케이스 처리]처럼 같은 원칙(추정 금지)을 여러 섹션에 걸쳐 다른 표현으로 반복 명시했다. 무료 모델은 한 번의 지시를 프롬프트 뒷부분까지 계속 지키지 못하는 경우가 있어, 짧은 지시 1개에 의존하지 않고 관련 규칙을 섹션별로 중복 배치해 누락 확률을 낮췄다.
2. **Few-shot 예시로 추론을 대체**: v2(4.2)처럼 "먼저 사실 확인 → 누락 질문 추출 → 형식 출력" 단계를 설명으로만 지시하는 대신, 5장의 3개 Few-shot 예시(정상/모호/빈 입력)를 그대로 프롬프트에 포함시켜 모델이 새로 추론하지 않고 패턴을 그대로 모방하게 했다. 무료 모델일수록 설명형 지시보다 예시 기반 모방이 더 안정적으로 작동한다.
3. **출력 자체에 자기 점검 규칙을 내장**: [엣지케이스 처리]에 "정보가 부족할 경우: 추정하지 말고 확인 항목을 질문"이라는 조건 분기를 프롬프트 안에 직접 넣어, 모델이 매 응답 전에 "이 항목이 메일에 있었는가?"를 스스로 체크하도록 강제했다. 별도의 검증 단계(2차 호출)를 추가할 API 비용 여유가 없으므로, 검증 로직 자체를 1차 프롬프트 안에 압축한 것이다.
4. **사람의 최종 확인 절차를 프로세스에 포함**: 무료판은 반복 검증 비용이 없으므로, REMI의 출력에서 (확인 필요) 태그가 붙은 항목은 담당자가 반드시 육안으로 재확인하도록 운영 절차화했다(8.6의 미확인 사항 분리 규칙과 연결). 즉 "모델이 100% 정확할 것"을 전제하지 않고, 태그 기반으로 사람이 개입할 지점을 미리 설계해둔 것이 무료 모델 사용 시 품질을 유지하는 핵심 전략이다.

### 10.2 업무 정책이 변경될 때, 환각을 줄이기 위한 운영 전략

메일 기반 요구사항 정리 과업은 실무에서 정책(예: 결제 정책, 보안 정책, 승인 절차)이 자주 바뀌는 영역이다. 정책 변경 상황에서 환각을 억제하기 위해 채택한 운영 전략은 다음 3단계다.

1. **프롬프트에 정책을 하드코딩하지 않는다**: 4.3 최종 프롬프트에는 특정 정책값(예: "PG사는 토스페이먼츠다")을 넣지 않았다. 정책은 항상 그때그때의 메일/대화 입력으로만 주입되게 하고, 프롬프트 자체는 "판단 규칙"(추정 금지, 확인 필요 태그)만 담당하게 분리했다. 정책이 바뀌어도 프롬프트를 다시 만들 필요가 없다.
2. **"확인 필요" 태그를 정책 변경의 안전판으로 사용한다**: 8.6에서 v1.3에 추가한 "미확인 사항은 반드시 (확인 필요) 태그로 분리 출력" 규칙이 바로 이 역할을 한다. 정책이 바뀌는 시점에는 필연적으로 "이전 정책이 아직 유효한지" 애매한 구간이 생기는데, 이때 REMI가 애매한 부분을 자동으로 확정 짓지 않고 무조건 미확인 사항으로 넘기도록 강제해, 정책 변경기의 환각 위험을 구조적으로 줄인다.
3. **범위 외 항목을 별도 섹션으로 격리한다**: Turn 5(6장)에서 보듯, 정책 변경(결제 취소/환불 제외)이 들어오면 [범위 외 항목] 섹션으로 옮기고 미확인 사항에서는 제거하는 방식을 프롬프트 규칙으로 정의했다(4.3 [엣지케이스 처리]). 이렇게 하면 "정책상 제외된 것"과 "아직 확인이 안 된 것"이 섞이지 않아, 정책 변경 이력 자체가 환각의 원인이 되는 것을 막는다.

### 10.3 10턴 이상 대화에서 문맥 단절 문제를 해결하기 위한 프롬프트 구조 개선

부록 A(8.5) A-Turn 4에서 실제로 "웹 전용" 조건이 이후 턴에서 "웹/앱 모두"로 잘못 서술되는 문맥 단절이 발생했다. 이를 해결하기 위해 프롬프트 구조를 다음과 같이 바꾸었다(8.6, v1.1→v1.2).

- **"업데이트" 태그로 변경 이력을 명시적으로 남긴다**: Turn 2, 6, 9(6장)처럼 이전 확인 사항이 갱신될 때는 반드시 `[기능 요구사항] (업데이트)`처럼 태그를 붙이도록 5장 Few-shot 예시에 패턴을 추가했다. 모델이 "새로 알게 된 정보"와 "이전부터 있던 정보"를 같은 취급으로 뭉뚱그리지 않고, 어떤 항목이 바뀌었는지 매 턴 출력에 드러내게 한 것이다.
- **이전 확인 사항을 누적해서 다시 출력하게 한다**: v1.2부터는 매 턴 결과에 "이번 턴에 새로 들어온 정보 + 지금까지 확정된 정보"를 함께 출력하도록 규칙을 두었다. 이전 턴 결과를 모델의 암묵적 기억에만 의존하지 않고, 매번 명시적으로 다시 서술하게 함으로써 중간에 조건이 누락되는 사고를 줄였다.
- **범위 외 항목과 미확인 사항을 분리해 상태를 고정한다**: Turn 5→Turn 10까지 [범위 외 항목] 섹션이 최종 문서(Turn 10)까지 그대로 유지되는 것도 같은 원리다. "이미 결론이 난 것"과 "아직 열려 있는 것"을 항상 별도 섹션으로 유지해, 턴이 길어져도 어느 정보가 확정 상태인지 다시 물어보지 않아도 알 수 있게 했다.
- **10턴 검증**: 8.3의 최종 검증 결과표는 이 구조 개선이 실제로 10턴 전체에서 조건 누락 없이 동작했는지를 확인한 기록이다.
